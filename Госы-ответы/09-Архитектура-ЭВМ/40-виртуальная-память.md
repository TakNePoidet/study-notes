## Развёрнутый ответ

### Зачем нужна виртуальная память

**Виртуальная память** — это механизм, дающий каждому процессу свое непрерывное адресное пространство, не зависящее от физического. Без него процессы делили бы реальную RAM напрямую, и каждый мог бы читать/писать память другого — никакой изоляции.

Виртуальная память решает сразу несколько задач:

- **изоляция процессов** — один процесс не видит память другого;
- **защита памяти** — страницы можно пометить только для чтения, только для исполнения, недоступными;
- **удобство** — каждый процесс думает, что у него непрерывный диапазон от 0 до огромного числа;
- **swap** — виртуальной памяти может быть больше, чем физической, лишнее уходит на диск;
- **shared mappings** — несколько процессов могут разделять общие страницы (библиотеки, IPC);
- **mmap файлов** — файлы проецируются в адресное пространство, ОС сама управляет загрузкой страниц;
- **copy-on-write при fork** — изначально процессы делят страницы, копирование при записи.

### Страничная организация

Виртуальная память делится на **страницы** фиксированного размера, обычно **4 КБ**. Физическая память тоже делится на **рамки** (page frames) того же размера. Виртуальная страница может быть отображена в любую физическую рамку.

Виртуальный адрес делится на две части:

```
Virtual addr (48 бит на x86-64):
| VPN (virtual page number, 36 бит) | offset (12 бит) |
                  │
                  ▼
            Page table → PPN (physical page number)
                  │
                  ▼
Physical addr:    | PPN | offset |
```

Старшая часть (VPN) переводится через таблицы страниц в номер физической рамки (PPN), младшая (offset) остаётся без изменений.

### Многоуровневые таблицы страниц

Если бы таблица была плоской, для 48-битного адресного пространства нужно было бы 2^36 записей на процесс — много гигабайт. Поэтому используют **многоуровневые таблицы**: на x86-64 — 4 или 5 уровней (PML4, PDPT, PD, PT). Большая часть таблиц «отсутствует», и место экономится.

Аппаратно это поддерживает **MMU**. На каждом обращении к памяти MMU «гуляет» по таблицам — это называется **page walk**.

### TLB — кэш переводов

Page walk при каждом обращении был бы убийственно медленным. Поэтому в CPU есть **TLB (Translation Lookaside Buffer)** — небольшой кэш переводов «виртуальная страница → физическая». TLB обычно содержит сотни записей и обновляется по мере работы программы.

**TLB miss** — промах TLB, нужен page walk. Это десятки наносекунд.

**TLB shootdown** — при изменении таблиц страниц нужно инвалидировать TLB на всех ядрах. Это дорогая операция.

### Page fault

Если виртуальная страница не отображена в физическую (нет в таблицах или нет в RAM), процессор генерирует исключение **page fault**. Ядро обрабатывает его:

1. Решает, валидный ли адрес (или это segmentation fault).
2. Определяет, нужно ли загрузить страницу с диска (**major fault**, медленно) или просто выделить новую страницу в памяти (**minor fault**, быстро).
3. Обновляет таблицу страниц.
4. Возвращает управление инструкции, которая повторяется и теперь успешно завершается.

Major faults — это I/O на диск, могут занимать миллисекунды. Minor faults — обычно микросекунды.

### Замещение страниц

Когда RAM заполнена, ОС должна выбрать страницу-«жертву» для вытеснения. Алгоритмы:

- **FIFO** — простой, но не учитывает фактическое использование.
- **LRU (Least Recently Used)** — наименее давно использованная. Идеален в теории, но дорог в реализации.
- **LFU (Least Frequently Used)** — наименее часто используемая.
- **Clock (second-chance)** — приближение LRU через кольцевой обход.
- **Working Set** — учитывает «рабочее множество» процесса.
- **2Q, CAR, ARC** — современные адаптивные алгоритмы.

В Linux используется приближение LRU с двумя списками (active/inactive) и сложными эвристиками; в новых версиях — **MGLRU (Multi-Generational LRU)**.

### Файл подкачки (swap)

В Linux — swap-раздел или файл (`mkswap`, `swapon`). В Windows — `pagefile.sys`.

Когда страница выбирается на вытеснение, её содержимое записывается в swap, и физическая рамка освобождается. При повторном обращении возникает major fault, страница загружается обратно.

Слишком активная подкачка — **thrashing** — это катастрофа для производительности. Лекарство — добавить RAM или ограничить рабочее множество.

### Дополнительные техники

**Demand paging.** Страница загружается только при первом обращении (lazy loading). Это уменьшает время старта программы.

**Copy-on-write (COW).** При `fork()` родитель и потомок делят все страницы памяти, помеченные как read-only. При попытке записи возникает page fault, ядро делает копию страницы только для пишущего процесса. Это экономит память и ускоряет fork.

**mmap файла.** Файл проецируется в адресное пространство процесса. Чтение/запись по этим адресам обращается к страницам, которые ядро лениво загружает с диска. Это удобно для больших файлов, БД, общих библиотек.

**Shared memory.** Через `shm_open` или `mmap` с MAP_SHARED несколько процессов могут разделять страницы. Самая быстрая IPC.

**Huge pages.** Страницы 2 МБ или 1 ГБ вместо 4 КБ. Снижают число TLB-промахов, важно для БД, JVM, ML. В Linux: `transparent_hugepages` или явное запрашивание через `madvise(MADV_HUGEPAGE)`.

### Защита и ASLR

Виртуальная память даёт инструмент защиты:

- **Битовые флаги страниц**: r/w/x. **NX bit** запрещает выполнение данных, что блокирует классические атаки переполнения буфера.
- **ASLR (Address Space Layout Randomization)** — рандомизация расположения сегментов (стек, heap, библиотеки) при каждом запуске. Усложняет эксплойты.
- **KPTI (Kernel Page Table Isolation)** — отдельные таблицы страниц для kernel и user. Введено как митигация против атаки Meltdown в 2018.
- **SMEP / SMAP** — аппаратные средства запрета исполнения user-кода и доступа к user-памяти из ядра.

### Производительность

Виртуальная память сама по себе создаёт накладные расходы, и хорошие приложения учитывают это:

- **TLB-локальность** — предсказуемые шаблоны доступа держат TLB заполненным.
- **Huge pages** для приложений с большим working set (БД, JVM).
- **Prefetching** — CPU и ОС угадывают, какие страницы нужны заранее.
- **NUMA-awareness** — в системах с несколькими NUMA-узлами доступ к локальной памяти быстрее, чем к удалённой.

### Связь с виртуализацией

В виртуальных машинах добавляется **двухуровневая трансляция**: гостевая виртуальная → гостевая физическая → хостовая физическая. Аппаратная поддержка — **EPT (Intel)** и **NPT (AMD)** — Extended/Nested Page Tables. Без них виртуализация была бы крайне медленной.

В контейнерах виртуальной памяти как таковой нет — контейнеры используют ядро хоста, но имеют свои **namespaces** и **cgroups** для изоляции.

### Что важно сказать в итоге

Виртуальная память — это аппаратно-программный механизм, дающий каждому процессу своё адресное пространство, изолированное от других. Реализуется через страничную организацию (страницы 4 КБ), многоуровневые таблицы страниц, MMU и TLB. При нехватке физической памяти ОС вытесняет страницы в swap. Дополнительные техники — demand paging, COW при fork, mmap файлов, shared memory, huge pages — расширяют возможности. Защита (NX bit, ASLR, KPTI, SMEP/SMAP) защищает от эксплойтов. В виртуализации добавляется второй уровень трансляции (EPT/NPT). Виртуальная память — это фундамент современной многозадачности и безопасности.
