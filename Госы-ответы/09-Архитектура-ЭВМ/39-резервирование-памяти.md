## Развёрнутый ответ

### Зачем разделять резервирование и выделение

В современной ОС адресное пространство процесса огромно — 256 ТБ на x86-64. При этом физической памяти всегда сильно меньше. Возникает разумная идея: **сначала зарезервировать виртуальный диапазон**, чтобы никто другой его не получил, а **физическую память выделять по мере фактической необходимости**.

Это даёт несколько преимуществ:

- можно «застолбить» большой непрерывный диапазон адресов, не платя за него памятью;
- стек потока резервирует мегабайт, а физически расходует только текущую глубину рекурсии;
- алгоритмы с разреженной памятью (большие хеши, sparse arrays) работают без огромного расхода RAM;
- виртуальные машины и JIT-компиляторы могут резервировать большие диапазоны под код, потом коммитить по мере необходимости.

В Windows этот подход формализован через два этапа: **MEM_RESERVE** (зарезервировать) и **MEM_COMMIT** (закоммитить, обязать ОС обеспечить физическую страницу).

### Состояния страницы в Windows

В Windows каждая страница виртуального адресного пространства может находиться в одном из состояний:

| Состояние | Описание |
|-----------|----------|
| **Free** | Адрес не используется, можно зарезервировать |
| **Reserved** | Виртуальный диапазон зарезервирован, физически не выделен |
| **Committed** | Выделена физическая страница (RAM или место в page file) |
| **Guard page** | Сигнализирует stack-overflow при касании |

Каждая закоммиченная страница имеет **атрибуты защиты**: `PAGE_NOACCESS`, `PAGE_READONLY`, `PAGE_READWRITE`, `PAGE_EXECUTE`, `PAGE_EXECUTE_READ`, `PAGE_EXECUTE_READWRITE`. Это позволяет, например, делать страницы только для чтения, или только для исполнения (важно для безопасности).

### API VirtualAlloc

Главная функция работы с памятью на низком уровне:

```c
LPVOID p = VirtualAlloc(
    NULL,                       // адрес (NULL — система выбирает)
    16 * 1024 * 1024,           // 16 MB
    MEM_RESERVE,                // только резерв, без выделения
    PAGE_NOACCESS);             // обращение запрещено

// Позже, по мере необходимости, коммитим кусок
LPVOID c = VirtualAlloc(
    p,                          // указываем уже зарезервированный адрес
    64 * 1024,                  // 64 KB
    MEM_COMMIT,                 // теперь точно выделить
    PAGE_READWRITE);            // и сделать читаемым/записываемым

// Освобождение
VirtualFree(c, 64 * 1024, MEM_DECOMMIT);  // вернуть физические страницы
VirtualFree(p, 0, MEM_RELEASE);           // снять резерв полностью
```

Можно одновременно резервировать и коммитить: `MEM_RESERVE | MEM_COMMIT`. Часто этот вариант и используется для обычных аллокаций.

Для другого процесса есть **VirtualAllocEx** — позволяет выделять память в чужом процессе (этим пользуются отладчики и иногда вредоносные программы).

### Стек потока

Каждый поток имеет свой стек. По умолчанию Windows **резервирует 1 МБ** виртуального адресного пространства под стек потока, **фиксируя** (commit) только первые страницы. По мере роста стека (вызовы функций, рекурсия) ОС автоматически расширяет коммит.

В конце зарезервированной области стоит **guard page** — особая страница, при касании к которой ОС бросает исключение. Это позволяет аккуратно расширять стек и обнаруживать stack overflow.

```c
HANDLE t = CreateThread(NULL,
    /*stack initial commit*/ 16 * 1024,
    proc, arg,
    STACK_SIZE_PARAM_IS_A_RESERVATION,
    NULL);
```

Для специфических задач (deep recursion) можно увеличить размер резерва до десятков мегабайт.

### Куча и менеджеры памяти

`VirtualAlloc` — это низкоуровневое API; работает с целыми страницами (4 КБ). Для обычных аллокаций используют **кучу (heap)**:

- **HeapAlloc / HeapFree** — куча процесса по умолчанию или собственная куча через `HeapCreate`.
- **CRT malloc / free**, `new / delete` в C++ — обёртки над кучей.
- **Process heaps** — несколько куч на процесс, разные для разных задач.
- **Pool allocators** — для частых однотипных объектов, дают лучшую производительность.

Менеджеры памяти под капотом всё равно используют `VirtualAlloc` для получения больших блоков от ОС.

### Файл подкачки и commit limit

Windows ограничивает суммарный коммит всех процессов суммой:

```
commit limit = физическая RAM + размер pagefile.sys
```

Если приложения суммарно закоммитили больше, чем есть RAM, часть «холодных» страниц вытесняется в файл подкачки **pagefile.sys**. При обращении к ним возникает страничная ошибка, и страница загружается обратно.

В диспетчере задач можно посмотреть две метрики процесса:

- **Commit Size** — сколько виртуальной памяти процесс закоммитил;
- **Working Set** — сколько физических страниц процесс реально использует сейчас.

Превышение commit limit приводит к ошибкам выделения и тормозам (swap thrashing).

### Аналоги в Linux

Linux работает похоже, но с другими понятиями.

**mmap.** Главная функция для работы с памятью на низком уровне:

```c
void *p = mmap(NULL, size, PROT_READ | PROT_WRITE,
               MAP_ANONYMOUS | MAP_PRIVATE, -1, 0);
```

Создаёт анонимный приватный маппинг — это и есть «резервирование + коммит» в одном вызове.

**Overcommit.** Linux позволяет «обещать» больше памяти, чем есть. Реальное выделение происходит при первом обращении (демандируемая страница). Если физическая память кончилась, срабатывает **OOM-killer** — убивает «жирный» процесс.

**MAP_NORESERVE.** Можно явно сказать «не резервировать swap».

**mprotect.** Изменение атрибутов защиты страниц.

**madvise(MADV_DONTNEED).** Сказать ядру, что данные больше не нужны — оно может освободить физические страницы.

### Где это пригодится

- **JIT-компиляторы** (JVM, V8, .NET). Резервируют большие диапазоны под генерируемый код, постепенно коммитят страницы.
- **Стек** для глубоких рекурсий — можно увеличить резерв до сотен мегабайт.
- **Sparse arrays** — массивы с разреженной заполненностью, виртуальный размер большой, реальный — маленький.
- **Виртуальные машины и эмуляторы** — выделяют гостевую память как один большой диапазон.
- **Memory-mapped files** — проекция файла в память, удобно для больших файлов.

### Что важно сказать в итоге

В Windows резервирование памяти происходит в два этапа: **MEM_RESERVE** забирает виртуальный диапазон, **MEM_COMMIT** обязывает ОС обеспечить физические страницы. Главный API — `VirtualAlloc`. Это позволяет «застолбить» большие диапазоны без расхода памяти и коммитить по мере необходимости. Стек потока резервирует 1 МБ по умолчанию, коммитит постранично. Над низким уровнем работают кучи (HeapAlloc) и менеджеры (malloc, new). Commit limit определяется суммой RAM и pagefile. В Linux аналог — `mmap` с overcommit. Эти механизмы — основа эффективной работы с памятью в JIT, эмуляторах, VM и больших структурах данных.
