## Развёрнутый ответ

### Сущность и принципы реляционной модели

Реляционная модель была сформулирована Эдгаром Коддом в 1970 году и стала основой современной индустрии баз данных. Её идея в том, что **вся информация представляется в виде таблиц (отношений), а связи между объектами выражаются не через указатели, а через значения общих атрибутов**.

Это даёт сразу несколько важных свойств:

- **декларативность** — мы говорим, что нам нужно получить, а как — решает оптимизатор;
- **независимость данных** от приложений — изменение физической организации не ломает запросы;
- строгий **математический фундамент** в виде реляционной алгебры и реляционного исчисления, на которых построен SQL.

Кодд позже сформулировал 12 правил, описывающих, какой должна быть «настоящая» реляционная СУБД. Главные из них — все данные представлены только в виде значений в таблицах; есть гарантированный доступ к любому значению по комбинации «таблица + ключ + столбец»; работа с NULL единообразна; язык запросов поддерживает определение схемы, манипуляцию данными, права доступа и транзакции.

### Базовые понятия

**Отношение (relation)** — это таблица, представляющая собой множество однородных строк. Слово «множество» здесь принципиальное: в отношении не должно быть двух одинаковых строк, и порядок строк не значим.

**Кортеж (tuple)** — строка таблицы, набор значений по всем атрибутам.

**Атрибут** — столбец, именованная характеристика. У каждого атрибута есть имя и тип.

**Домен** — это множество допустимых значений атрибута, то есть его «тип со смыслом»: например, целое положительное число, ИНН, email, дата. Технически домен реализуется через типы данных СУБД и ограничения CHECK.

Дополнительно:

- **степень отношения** — количество атрибутов (столбцов);
- **кардинальность** — количество кортежей (строк);
- **схема отношения** — набор имён и доменов атрибутов.

Важное свойство реляционных таблиц: **значения атрибутов атомарны** (первая нормальная форма). Никаких списков через запятую внутри одной ячейки.

### Логические связи между таблицами

Связи в реляционной модели — это всегда **связи по значениям**, а не по указателям. Если у меня есть таблица заказов и таблица пользователей, я не храню в строке заказа «указатель» на строку пользователя — я храню значение его идентификатора (`user_id`), и СУБД при необходимости найдёт нужную строку.

Типы связей:

- **один к одному (1:1)** — реализуется через UNIQUE-ограничение на внешний ключ или через общий первичный ключ;
- **один ко многим (1:N)** — внешний ключ хранится на стороне «многих» (заказы → пользователь);
- **многие ко многим (N:M)** — выносится в отдельную таблицу-связку, у которой составной первичный ключ из двух внешних.

Например, связь «студент учится у многих преподавателей, преподаватель учит многих студентов» реализуется так:

```sql
CREATE TABLE students (id SERIAL PRIMARY KEY, name VARCHAR(100));
CREATE TABLE teachers (id SERIAL PRIMARY KEY, name VARCHAR(100));
CREATE TABLE student_teacher (
  student_id INT REFERENCES students(id),
  teacher_id INT REFERENCES teachers(id),
  PRIMARY KEY (student_id, teacher_id)
);
```

### Виды и назначение ключей

Ключ в реляционной модели — это атрибут или набор атрибутов, обладающий определёнными свойствами.

**Суперключ** — любой набор атрибутов, который уникально идентифицирует кортеж. По сути, это «избыточный» ключ — в него можно включить лишние столбцы, и он останется уникальным.

**Кандидатный ключ** — минимальный суперключ, то есть такой, из которого нельзя выбросить ни один атрибут без потери уникальности. У одной таблицы может быть несколько кандидатных ключей.

**Первичный ключ (Primary Key, PK)** — один из кандидатных ключей, выбранный для основной идентификации строк. Первичный ключ должен быть NOT NULL и UNIQUE, и обычно по нему автоматически строится кластерный или обычный индекс. PK используется как точка ссылки для внешних ключей.

**Альтернативный ключ** — оставшиеся кандидатные ключи, не выбранные в качестве первичного. Обычно реализуются через UNIQUE-ограничение.

**Составной ключ** — ключ, состоящий из нескольких атрибутов. Полезен, когда сущность естественно определяется комбинацией значений (например, пара студент+курс в таблице оценок).

**Суррогатный ключ** — искусственно введённый ключ, не несущий бизнес-смысла: автоинкремент (SERIAL, IDENTITY), UUID, snowflake-id. Суррогатные ключи проще: они стабильны (не меняются при изменении бизнес-данных), компактны и не создают проблем при рефакторинге. На практике большинство современных схем использует именно их.

**Естественный ключ** — ключ, основанный на бизнес-смысле (ИНН, артикул, email). Использовать можно, но осторожно: бизнес-сущности могут меняться, и тогда придётся менять идентификатор повсюду.

**Внешний ключ (Foreign Key, FK)** — атрибут (или набор), значения которого должны соответствовать значениям первичного ключа в другой таблице (или той же). Внешний ключ выражает связь и обеспечивает **ссылочную целостность**: нельзя сослаться на несуществующую строку.

### Пример со всеми типами ключей

```sql
CREATE TABLE departments (
  dept_id   SERIAL PRIMARY KEY,        -- суррогатный первичный ключ
  code      VARCHAR(10) UNIQUE,        -- альтернативный (кандидатный) ключ
  name      VARCHAR(100) NOT NULL
);

CREATE TABLE employees (
  emp_id    SERIAL PRIMARY KEY,        -- суррогатный PK
  email     VARCHAR(255) UNIQUE,       -- альтернативный ключ
  name      VARCHAR(100) NOT NULL,
  dept_id   INT REFERENCES departments(dept_id) ON DELETE SET NULL
);
```

Поле `dept_id` в таблице сотрудников — это внешний ключ. Поведение `ON DELETE SET NULL` означает, что при удалении отдела поле dept_id у сотрудников будет установлено в NULL; альтернативы — `CASCADE` (удалить и сотрудников), `RESTRICT` (запретить удаление, если есть ссылки), `NO ACTION` (по умолчанию похоже на RESTRICT).

### Целостность данных

Реляционная модель различает несколько видов целостности.

**Сущностная целостность** — каждая строка идентифицируема через первичный ключ; PK не может содержать NULL и должен быть уникальным. Это база реляционной модели.

**Ссылочная целостность** — внешний ключ либо ссылается на существующую строку, либо равен NULL. СУБД сама проверяет это при вставке, обновлении и удалении.

**Доменная целостность** — значения атрибутов соответствуют их типу и ограничениям. Реализуется через типы данных, NOT NULL, CHECK, UNIQUE.

**Бизнес-целостность (пользовательская)** — дополнительные правила предметной области, например «количество не может быть отрицательным» или «дата начала не позже даты конца». Реализуется через CHECK-ограничения или триггеры.

### Что важно сказать в итоге

Реляционная модель строится на простой и при этом мощной идее: всё — это таблицы из строк и столбцов, связи между ними — через значения ключей, а целостность данных гарантируется самой СУБД. Понимание ролей разных типов ключей (PK, FK, кандидатный, суррогатный) и видов целостности — это фундамент, на котором строится грамотное проектирование любой реляционной базы.
