# 6. Характеристики технологий ADO, OLEDB, ODBC

## План ответа

1. Зачем нужны прослойки доступа к данным.
2. ODBC — первый стандарт, низкоуровневый C-API.
3. OLE DB — COM-надстройка от Microsoft, шире ODBC, умеет нереляционные источники.

## Развёрнутый ответ

### Зачем нужны прослойки доступа к данным

Когда приложение хочет работать с базой данных, ему нужен механизм передачи запросов и получения результатов. У каждой СУБД свой нативный протокол: один у PostgreSQL, другой у Oracle, третий у MS SQL. Если приложение поддерживает несколько баз, привязка к каждой реализации делает код нерасширяемым.

Чтобы абстрагироваться от конкретной СУБД, в индустрии появились **прослойки доступа к данным**. Идея простая: приложение разговаривает с прослойкой через стандартный API, а прослойка через драйверы общается с конкретной СУБД. Меняем СУБД — меняем драйвер, код приложения остаётся прежним.

ODBC, OLE DB и ADO — это три исторически связанные технологии Microsoft, образующие иерархию от низкого уровня к высокому.

### ODBC (Open Database Connectivity)

ODBC был разработан Microsoft и SQL Access Group в 1992 году и стал **первым массовым стандартом** доступа к реляционным БД. Архитектура состоит из четырёх слоёв: приложение → ODBC Driver Manager → драйвер конкретной СУБД → база.

ODBC — это **низкоуровневый C-API**: программист на C/C++ вызывает функции вроде `SQLAllocHandle`, `SQLConnect`, `SQLExecDirect`, `SQLFetch`. Источники данных описываются специальным конфигурационным идентификатором — **DSN (Data Source Name)**, в котором задаются строка подключения, логин, опции драйвера.

Основные характеристики:

- кросс-платформенный — работает на Windows, Linux, macOS;
- ориентирован только на **реляционные источники** (SQL);
- низкоуровневый и поэтому весьма гибкий, но громоздкий в использовании;
- сегодня широко применяется как «универсальный шлюз» — ODBC-драйверы есть у всех СУБД.

В современных стеках ODBC живёт под капотом: например, библиотека `pyodbc` в Python или ODBC-подключения в BI-системах вроде Tableau и Power BI используют именно ODBC-драйверы.

### OLE DB

OLE DB появился в 1996 году как часть Microsoft Data Access Components (MDAC). Это уже не C-API, а набор **COM-интерфейсов**. По задумке OLE DB должен был быть мощнее и универсальнее ODBC.

Ключевая идея — OLE DB умеет работать **не только с реляционными источниками**: через OLE DB-провайдеров можно подключаться к Excel-файлам, текстовым файлам, Active Directory, почте Exchange, индексам файловой системы и многому другому. Каждому типу источника соответствует свой провайдер.

При этом существует **OLE DB Provider for ODBC**, который позволяет использовать ODBC-драйверы через OLE DB-интерфейс. Это сделало OLE DB совместимым с уже накопленной экосистемой ODBC.

Из-за того что OLE DB основан на COM, он привязан к Windows и в основном использовался из C++/Delphi. API считается сложным, требует понимания COM-объектов, интерфейсов, ссылок и подсчёта реликов.

### ADO (ActiveX Data Objects)

ADO появился одновременно с OLE DB и был задуман как **высокоуровневая обёртка** над ним для скриптовых сред — VB, VBA, ASP, VBScript, JScript. ADO скрывает сложность COM-интерфейсов OLE DB за простыми объектами и свойствами.

Основные объекты ADO:

- **Connection** — соединение с базой;
- **Command** — подготовленный запрос с параметрами;
- **Recordset** — набор записей, можно навигировать, фильтровать, обновлять;
- **Parameter** — параметр для запроса;
- **Error** — описание ошибок.

Пример использования из классического ASP / VBScript:

```vbscript
Dim conn, rs
Set conn = CreateObject("ADODB.Connection")
conn.Open "Provider=SQLOLEDB;Data Source=server;Initial Catalog=db;User Id=sa;Password=***"

Set rs = conn.Execute("SELECT id, name FROM users WHERE active = 1")
Do While Not rs.EOF
  Response.Write rs("name") & "<br>"
  rs.MoveNext
Loop
rs.Close: conn.Close
```

ADO быстро стал популярным в 90-х и начале 2000-х, особенно в веб-разработке на ASP и десктоп-приложениях на VB6.

### Иерархия и взаимосвязь

Схематически связь между технологиями выглядит так:

```
Приложение → ADO → OLE DB → (через провайдер) → СУБД
                                ↘ через OLE DB Provider for ODBC → ODBC → СУБД
```

ADO — самый высокий уровень: удобен, но привязан к OLE DB. OLE DB — средний: универсальный, может работать с нереляционными источниками. ODBC — самый низкий: универсальный, простой, но громоздкий.

### Современное состояние

Сегодня большинство этих технологий считаются **устаревающими**, но не везде заменены.

- **ODBC** живёт и используется как универсальный коннектор: BI-инструменты, Python, Tableau, Power BI постоянно работают через ODBC-драйверы.
- **OLE DB** Microsoft в 2011 году объявил устаревшим, заменив на ODBC. Но в 2017-м его реанимировали — выпустили MSOLEDBSQL для SQL Server. Полностью OLE DB не ушёл, но новых разработок на нём почти нет.
- **ADO** заменён на **ADO.NET** в .NET-экосистеме. Это уже отдельная технология: `SqlConnection`, `SqlCommand`, `SqlDataReader`, `DataSet`, `DataTable`. ADO.NET — основа для ORM (Entity Framework, Dapper).

Из других экосистем:

- В Java стандартом доступа к БД является **JDBC** — концептуальный аналог ODBC.
- В Python — **DB-API 2.0** (драйверы `psycopg2`, `pymysql`, `pyodbc`).
- В Node.js — нативные драйверы (`pg`, `mysql2`) или ORM (Prisma, TypeORM, Sequelize).

### Сравнительная таблица

| Свойство | ODBC | OLE DB | ADO |
|----------|------|--------|-----|
| Тип API | C, низкоуровневый | COM | COM, высокоуровневый |
| Источники | Реляционные | Любые (через провайдеров) | Через OLE DB |
| Платформа | Кросс-платформа | Windows | Windows |
| Удобство | Низкое | Среднее | Высокое |
| Год появления | 1992 | 1996 | 1996 |
| Современное состояние | Поддерживается, активно используется | Устаревает | Устарел, заменён ADO.NET |

### Что важно сказать в итоге

ODBC, OLE DB и ADO — это три исторически связанные технологии доступа к данным от Microsoft, расположенные на разных уровнях абстракции. ODBC даёт универсальный низкоуровневый интерфейс к реляционным СУБД, OLE DB расширяет это на нереляционные источники через COM, а ADO предоставляет удобную обёртку для скриптовых сред. Сегодня их роль постепенно занимают современные API — ADO.NET, JDBC, нативные драйверы языков — но понимание иерархии этих технологий нужно для работы с легаси-системами и многими аналитическими инструментами, которые до сих пор работают через ODBC.
