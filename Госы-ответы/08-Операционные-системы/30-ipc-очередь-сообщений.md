## Развёрнутый ответ

### Зачем нужны IPC

**IPC (Inter-Process Communication)** — это механизмы обмена данными и координации между процессами. У каждого процесса своё адресное пространство, и просто так передать данные между ними нельзя — для этого нужны специальные средства.

IPC решает несколько задач:

- **передача данных** между процессами;
- **синхронизация** действий нескольких процессов;
- **управление** — родительский процесс отправляет команды дочерним;
- **реализация клиент-серверной модели** на одной машине (демоны и приложения).

### Виды IPC

В Unix/Linux есть несколько механизмов, и каждый подходит для своих задач.

| Механизм | Тип данных | Особенности |
|----------|-----------|-------------|
| **Pipe (анонимный)** | Поток байт | Только для родственных процессов (через fork) |
| **Named pipe (FIFO)** | Поток байт | Через файл в ФС, для любых процессов |
| **Сигнал** | Уведомление | Очень ограниченные данные (1 номер) |
| **Разделяемая память (SHM)** | Общая память | Самый быстрый, нужна отдельная синхронизация |
| **Семафор** | Счётчик | Для синхронизации, не передачи данных |
| **Очередь сообщений** | Дискретные пакеты | Асинхронно, с приоритетом |
| **Сокеты** | Поток или датаграммы | Универсальные, в т.ч. между машинами |
| **Memory-mapped file (mmap)** | Память + файл | Доступ к файлу как к памяти |

Из этого списка **очередь сообщений** — самостоятельный механизм для передачи **дискретных пакетов**, в отличие от pipe (поток байт) или shared memory (общая память без структуры).

### POSIX-очереди сообщений

POSIX-API более современный, появился в стандарте POSIX.1b. Очередь идентифицируется именем, начинающимся с `/`, и видна в файловой системе через `/dev/mqueue/`.

```c
#include <mqueue.h>

// Создать или открыть очередь
mqd_t mq = mq_open("/myq", O_CREAT | O_RDWR, 0644, NULL);

// Отправить сообщение с приоритетом
const char *msg = "hello";
mq_send(mq, msg, strlen(msg) + 1, /*prio=*/ 1);

// Получить сообщение
char buf[1024];
unsigned prio;
ssize_t n = mq_receive(mq, buf, sizeof(buf), &prio);

// Закрыть и удалить
mq_close(mq);
mq_unlink("/myq");
```

Атрибуты очереди (структура `mq_attr`): максимальное число сообщений, максимальный размер одного сообщения. Сообщения упорядочены **по приоритету**: высший приоритет получается первым.

**Особенности POSIX:**

- Имена в стиле файлов;
- Приоритеты сообщений;
- Поддержка **mq_notify** — уведомление через сигнал или поток при поступлении сообщения;
- Современный, понятный API.

### System V-очереди сообщений

System V — более старый стандарт. Очередь идентифицируется числовым ключом, обычно генерируемым через `ftok` (по имени файла и проектному идентификатору).

```c
#include <sys/msg.h>

// Получить ключ и создать очередь
key_t key = ftok("/tmp/somefile", 65);
int qid = msgget(key, 0644 | IPC_CREAT);

// Структура сообщения
struct msg { long type; char text[100]; } m = { 1, "hello" };

// Отправить
msgsnd(qid, &m, sizeof(m.text), 0);

// Получить (по типу — фильтр)
msgrcv(qid, &m, sizeof(m.text), 1, 0);

// Удалить очередь
msgctl(qid, IPC_RMID, NULL);
```

Управление и просмотр из shell:

```bash
ipcs -q                     # список System V очередей
ipcrm -q <ID>               # удалить
```

**Особенности System V:**

- Числовые ключи (через `ftok`);
- Каждое сообщение имеет «тип» — целое число, по которому можно фильтровать;
- Нет приоритетов как таковых, но фильтрация по типу даёт похожее поведение;
- Нет уведомлений;
- Старый и неинтуитивный API.

### Сравнение POSIX и System V

| Свойство | POSIX | System V |
|----------|-------|----------|
| API | `mq_*` | `msgget`, `msgsnd`, `msgrcv` |
| Имена | Файловый путь `/name` | Числовые ключи (`ftok`) |
| Приоритеты | Да, у каждого сообщения | Нет (только тип) |
| Уведомления | `mq_notify` через сигнал или поток | Нет |
| Просмотр | `/dev/mqueue/` | `ipcs -q` |
| Удобство | Современный | Устарел |

В новых проектах однозначно лучше использовать POSIX.

### Когда что использовать в IPC в целом

- **Pipe/FIFO** — если нужен простой поток данных между процессами на одной машине.
- **Shared memory + семафор или мьютекс** — когда нужен большой объём данных с максимальной скоростью, готовы сами заботиться о синхронизации.
- **Очередь сообщений** — асинхронный обмен короткими пакетами, с приоритетами.
- **Unix-сокеты** — современный универсальный механизм. Поддерживает передачу credentials, передачу файловых дескрипторов между процессами. Часто заменяет всё остальное.
- **TCP/UDP-сокеты** — когда процессы могут быть на разных машинах.
- **D-Bus** — стандарт IPC для desktop в Linux, удобен для системных и пользовательских сервисов.

### Безопасность IPC

Объекты IPC — это ресурсы, которые при правильной настройке доступны нужным пользователям, при неправильной — открыты всем.

- Права на объект IPC (`mq_open` принимает mode, аналогично файлам);
- **Изоляция через namespaces** в Linux: **IPC namespace** изолирует System V и POSIX очереди;
- В контейнерных средах (Docker, Kubernetes) объекты IPC по умолчанию изолированы между контейнерами.

### Современный контекст

В прикладной разработке вы редко используете System V IPC напрямую. Чаще встречается:

- **Pipe/Unix-сокеты** в shell-скриптах и системных утилитах;
- **Сетевые сокеты + JSON/Protobuf** для общения между сервисами;
- **Брокеры сообщений** (Apache Kafka, RabbitMQ, NATS) — это уже **сетевой уровень**, не системный IPC, но решают аналогичную задачу на распределённой системе;
- **Shared memory** — в высокопроизводительных приложениях (БД, ML-инференс).

### Что важно сказать в итоге

IPC — это механизмы взаимодействия процессов: pipes, сигналы, разделяемая память, семафоры, очереди сообщений, сокеты, memory-mapped files. **Очередь сообщений** — это асинхронный канал, по которому процессы обмениваются дискретными пакетами. Существуют два API: POSIX (современный, имена в стиле файлов, приоритеты, mq_notify) и System V (устаревший, числовые ключи, фильтрация по типу). На практике в новых проектах используют POSIX-очереди, Unix-сокеты или сетевые брокеры сообщений. Для системных задач — pipes, сигналы и shared memory остаются основой.
