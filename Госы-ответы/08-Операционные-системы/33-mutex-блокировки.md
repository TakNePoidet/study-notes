## Развёрнутый ответ

### Критическая секция и мьютекс

**Критическая секция** — это участок кода, в котором поток работает с разделяемым ресурсом. Если несколько потоков одновременно входят в критическую секцию, возникают **гонки данных (race conditions)** — результат становится непредсказуемым.

Пример без защиты:

```c
int counter = 0;

void *inc(void *arg) {
    for (int i = 0; i < 100000; i++) {
        counter++;          // ГОНКА!
    }
    return NULL;
}
```

Если два потока выполнят это одновременно, итоговое значение будет меньше 200000: операция `counter++` не атомарна — это **load → increment → store**, и между шагами другой поток может «вклиниться».

**Mutex (mutual exclusion)** — примитив, гарантирующий, что в критической секции одновременно находится не более одного потока:

```c
pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;

void *inc(void *arg) {
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&m);
        counter++;                   // безопасно
        pthread_mutex_unlock(&m);
    }
    return NULL;
}
```

Свойства правильного решения:
- **взаимное исключение** — один поток в секции;
- **прогресс** — если никто не держит, очередной получает доступ;
- **ограниченное ожидание** — никто не ждёт бесконечно.

### Виды блокировок

| Тип | Описание | Когда применять |
|-----|----------|-----------------|
| **Mutex** | Один владелец, ожидающие усыпляются | Стандартные критические секции |
| **Spinlock** | Активное ожидание (busy-wait) | Очень короткая секция на нескольких ядрах |
| **Recursive mutex** | Один поток может захватить несколько раз | Когда внутри секции вызывается код, который тоже её захватывает |
| **RWLock (читатели-писатели)** | Много читателей или один писатель | Когда чтений намного больше, чем записей |
| **Adaptive mutex** | Сначала немного спинит, потом блокируется | Linux-оптимизация, среднее между mutex и spinlock |
| **Ticket / MCS lock** | FIFO-очередь, справедливость | Когда важна справедливость и нет голодания |
| **Futex** (Linux) | Быстрый user-space пути, в ядро — только при contention | Внутри pthread_mutex |
| **Семафор** | Счётчик, разрешает N одновременных | Ограничение числа одновременных операций |
| **Condition variable** | Ждать условие + mutex | Producer-Consumer, событийные модели |

### Атомарные операции

Иногда мьютекс — это слишком тяжело. Для простых случаев (счётчик, флаг) можно обойтись **атомарными операциями**, которые гарантируют неделимость на уровне CPU:

```c
// C11
#include <stdatomic.h>
atomic_int counter = 0;
atomic_fetch_add(&counter, 1);
```

```cpp
// C++
#include <atomic>
std::atomic<int> counter{0};
counter.fetch_add(1, std::memory_order_relaxed);
```

Атомарные операции реализуются специальными инструкциями CPU: `LOCK CMPXCHG`, `LOCK XADD` на x86. Они быстрее мьютексов, но решают только простые задачи. На атомарных операциях строятся **lock-free** структуры данных (Treiber stack, Michael-Scott queue) — сложные, но не имеющие классических блокировок.

### Проблемы

**Deadlock — взаимная блокировка.** Два потока ждут блокировки друг друга. Классический пример: поток A захватил lock1 и хочет lock2, поток B захватил lock2 и хочет lock1. Никто не движется.

Условия Coffman для deadlock:
1. **Взаимное исключение** — ресурс не разделяется.
2. **Удержание и ожидание** — поток держит ресурс и ждёт другого.
3. **Отсутствие вытеснения** — нельзя забрать ресурс силой.
4. **Циклическое ожидание** — поток A ждёт B, B ждёт C, C ждёт A.

Решения:
- **Единый порядок захвата** — все потоки берут lock1 перед lock2.
- **`try_lock` с откатом** — попробовать, не получилось — отпустить и заново.
- **Иерархия мьютексов** — каждому уровень, нельзя брать «вверх».
- **Lock-free алгоритмы**.

**Livelock.** Потоки реагируют друг на друга и не делают прогресса. Например, оба «вежливо уступают», и оба ничего не делают.

**Starvation.** Один поток никогда не получает блокировку из-за «жадности» других. Решение — справедливые блокировки (FIFO).

**Priority inversion.** Высокоприоритетный поток ждёт низкоприоритетного, который держит mutex. Известный кейс — миссия Mars Pathfinder в 1997 году, где это вызвало проблемы. Решение — **наследование приоритета** (priority inheritance).

**Convoy effect.** Очередь коротких задач выстраивается за одной долгой, общая производительность падает.

**False sharing.** Две переменные оказались в одной кэш-линии CPU, и обновление одной инвалидирует кэш для другой. Решается выравниванием.

### Рецепты безопасной работы

**RAII в C++.** Лучший способ не забыть `unlock` — RAII-обёртка:

```cpp
#include <mutex>
std::mutex m;
{
    std::lock_guard<std::mutex> g(m);  // lock в конструкторе
    // критическая секция
}                                      // unlock в деструкторе
```

Даже при exception unlock сработает.

**Захват нескольких мьютексов.** `std::lock` атомарно захватывает все, без deadlock:

```cpp
std::unique_lock<std::mutex> ul1(m1, std::defer_lock);
std::unique_lock<std::mutex> ul2(m2, std::defer_lock);
std::lock(ul1, ul2);  // безопасно
```

**Минимизировать критическую секцию.** Чем короче — тем меньше конкуренция и быстрее код.

**Не делать в критической секции I/O.** Никаких `printf`, `read`, сетевых запросов — это держит блокировку слишком долго.

### Когда какой инструмент

- **Очень короткая критическая секция, низкая конкуренция** — spinlock.
- **Стандартная длительная работа** — mutex (futex).
- **99% операций — чтение** — rwlock или seqlock.
- **Простой счётчик/флаг** — atomic.
- **Сложная синхронизация состояния** — condition variable + mutex или future/promise.
- **Высокоуровневые задачи** — channels (Go, Rust mpsc), акторы, async.

### Инструменты обнаружения проблем

- **ThreadSanitizer** (`-fsanitize=thread`) — обнаруживает гонки в рантайме.
- **Helgrind / DRD** (Valgrind) — анализ синхронизации.
- **lockdep** в ядре Linux — обнаружение потенциальных deadlock-ов.
- **perf**, **eBPF-tools** (например, `funclatency`) — измерение contention.

### Современный подход

Современная разработка старается избегать ручной синхронизации. Лучшие практики:

- **Иммутабельные структуры данных** — нет состояния, нет гонок.
- **Готовые concurrent-структуры**: `ConcurrentHashMap` в Java, `sync.Map` в Go, лок-free очереди.
- **Высокоуровневые модели** — async/await, акторы, channels.
- **Изоляция состояния** — каждый поток имеет своё, общение через сообщения.

Чем меньше разделяемого изменяемого состояния, тем меньше класс багов многопоточности.

### Что важно сказать в итоге

Mutex — это базовый примитив синхронизации, гарантирующий взаимное исключение в критической секции. Видов блокировок много: обычные мьютексы, spinlock, recursive, rwlock, adaptive, ticket, futex; всё это варианты под разные сценарии. Атомарные операции и lock-free алгоритмы — альтернатива блокировкам для простых случаев. Главные проблемы — deadlock, livelock, starvation, priority inversion, convoy, false sharing. Современный подход — минимизировать разделяемое состояние, использовать готовые concurrent-структуры, высокоуровневые модели (async, акторы, channels) и инструменты обнаружения (ThreadSanitizer, lockdep).
