# 31. Многопоточное программирование. Программные каналы

## План ответа

1. Что такое многопоточность и зачем она нужна.
2. Различия между процессом и потоком.
3. Преимущества и сложности многопоточности.

## Развёрнутый ответ

### Что такое многопоточность

**Многопоточность** — это выполнение нескольких потоков управления внутри одного процесса. Потоки одного процесса разделяют память, файловые дескрипторы, окружение, но имеют свои стеки и регистры. Это лёгкие сущности — создание потока в разы дешевле, чем создание процесса.

Многопоточность нужна для нескольких задач:

- **Параллелизм** — использование нескольких ядер CPU для ускорения вычислений.
- **Отзывчивость интерфейсов** — основной поток отвечает за UI, фоновые делают тяжёлую работу.
- **Параллельный I/O** — пока один поток ждёт ответа от сети или диска, другие работают.
- **Удобство архитектуры** — для серверов: один поток на каждое клиентское подключение (хотя сейчас это вытеснили async-модели).

### Процесс vs поток

| Свойство | Процесс | Поток |
|----------|---------|-------|
| Адресное пространство | Отдельное | Общее с другими потоками процесса |
| Создание | Дорогое (`fork`) | Дешёвое (`clone`/`pthread_create`) |
| Связь | Через IPC | Через общую память |
| Изоляция | Сильная | Нет |
| Цена ошибки | Падает один процесс | Падает весь процесс |

В Linux потоки реализованы через системный вызов `clone(2)` с флагами разделения ресурсов. По сути, это «лёгкие процессы», что хорошо для производительности.

### Преимущества и сложности

**Преимущества:**

- Использование нескольких ядер CPU без сложного IPC.
- Эффективная работа с I/O без блокирования UI.
- Логическое разделение задач (worker pool, producer-consumer).

**Сложности:**

- **Race conditions** — состояние гонки. Результат зависит от порядка выполнения потоков.
- **Deadlock** — взаимная блокировка двух или более потоков.
- **Livelock** — потоки реагируют друг на друга и не делают прогресса.
- **Starvation** — поток не получает ресурс из-за «жадности» других.
- **False sharing** — кэш-линии CPU инвалидируются из-за соседних переменных.
- **Сложность отладки** — баги многопоточности часто не воспроизводятся.

Поэтому к многопоточности подходят осторожно: меньше разделяемого состояния — меньше проблем.

### Программные каналы (pipes)

**Pipe** — это однонаправленный канал передачи данных между процессами. Технически — кольцевой буфер в ядре. Один процесс пишет в один конец, другой читает из другого.

**Анонимный pipe.** Создаётся через `pipe()` и работает только между родственными процессами (родитель и потомок через `fork`):

```c
int fd[2];
pipe(fd);                 // fd[0] — read, fd[1] — write

if (fork() == 0) {
    // Ребёнок: пишет
    close(fd[0]);
    write(fd[1], "hi\n", 3);
    close(fd[1]);
    exit(0);
} else {
    // Родитель: читает
    close(fd[1]);
    char buf[16];
    int n = read(fd[0], buf, sizeof(buf));
    close(fd[0]);
}
```

Анонимный pipe однонаправленный. Для двусторонней связи открывают пару pipe или используют `socketpair()`.

**Именованный pipe (FIFO).** Имеет имя в файловой системе и работает между любыми процессами:

```bash
mkfifo /tmp/myfifo
# Терминал 1:
echo "hello" > /tmp/myfifo
# Терминал 2:
cat /tmp/myfifo
```

FIFO — это не обычный файл, данные не сохраняются на диск. Это просто способ дать pipe имя, чтобы любой процесс мог открыть.

### Свойства pipes

- **Атомарность записи до PIPE_BUF байт** (обычно 4096) гарантирована: если несколько процессов одновременно пишут небольшие сообщения, они не перемешиваются.
- **Буфер ограничен** (обычно 64 КБ): при заполнении writer **блокируется**, пока кто-то не прочитает.
- **SIGPIPE при закрытии reader-а:** если все читатели закрыли свой конец, writer при попытке писать получает сигнал SIGPIPE (по умолчанию убивающий процесс).
- **EOF при закрытии writer-а:** если все писатели закрыли свой конец, reader получает 0 байт (как при достижении конца файла).

### Shell pipelines

Command pipeline в shell — это классический пример использования pipes:

```bash
cat access.log | grep 404 | awk '{print $7}' | sort | uniq -c | sort -rn | head
```

Shell создаёт цепочку процессов, соединённых анонимными pipe-ами. stdout одного становится stdin следующего. Это позволяет легко комбинировать утилиты и обрабатывать большие данные потоком, без необходимости держать всё в памяти.

### Многопоточность и pipes — Producer-Consumer

Классическая задача параллелизма — **Producer-Consumer**: один поток или процесс производит данные, другой потребляет. Pipe — естественная реализация этого паттерна на уровне ОС, но **внутри процесса** обычно используют:

- **мьютекс + условную переменную** + общую очередь;
- **lock-free queue**;
- **channel** (Go, Rust mpsc, Kotlin Channel) — высокоуровневая абстракция.

### Современные альтернативы

В современной разработке многопоточность часто прячут за более удобными абстракциями.

**Channels.** Go-горутины общаются через каналы:

```go
ch := make(chan int)
go func() { ch <- 42 }()
v := <-ch
```

**Async I/O / event loop.** Node.js, Python asyncio — не используют много потоков, а делают неблокирующие операции внутри одного цикла событий. Это часто эффективнее для I/O-задач.

**Coroutines / fibers.** Лёгкие задачи в user space (Go goroutines, Kotlin coroutines, C++ coroutines).

**Акторы.** Erlang, Akka, Orleans — модель, где каждый «актор» имеет свою почтовую очередь и обменивается сообщениями. Нет общего состояния, нет deadlock-ов.

**Pools.** ThreadPoolExecutor в Java/Python, Worker Pool в Go — пулы потоков, скрывающие низкоуровневую работу.

### Когда что использовать

- **Pipes** — линейная цепочка обработки на одной машине, классика shell.
- **Sockets** — двусторонняя связь, в том числе сетевая.
- **Очереди сообщений** — асинхронные пакеты с приоритетом.
- **Shared memory** — большой объём данных в пределах машины с самостоятельной синхронизацией.
- **Каналы / акторы** — современный код, где хочется избежать ручной синхронизации.

### Что важно сказать в итоге

Многопоточность — это выполнение нескольких потоков управления в одном процессе. Она нужна для параллелизма, отзывчивости UI и удобной архитектуры, но добавляет сложности: race conditions, deadlocks, starvation. **Программные каналы (pipes)** — однонаправленные потоки байт между процессами или потоками. Анонимные pipes работают между родственными процессами, именованные (FIFO) — между любыми. Pipes лежат в основе shell-pipelines и паттерна producer-consumer. В современной разработке низкоуровневая работа с потоками и pipes часто скрыта за абстракциями: каналы (Go), async/await, акторы, пулы потоков — это снижает риск ошибок синхронизации и упрощает код.
