## Развёрнутый ответ

### Что такое MapReduce

**MapReduce** — это модель распределённых вычислений, предложенная Google в 2004 году в статье Джеффри Дина и Санжая Гемавата. Это вторая «нога» оригинальной Hadoop-архитектуры (первая — HDFS), которая позволила обрабатывать огромные объёмы данных на кластере commodity-серверов.

Идея простая: разделить большой файл на куски, обработать каждый параллельно, потом сгруппировать промежуточные результаты по ключу и свести в финальный ответ. Программист пишет всего две функции — **map** и **reduce**. Всё остальное — планирование задач, распределение по узлам, обработка сбоев, перезапуск — берёт на себя фреймворк.

MapReduce — это **программная модель**, и Hadoop MapReduce — её конкретная реализация на Java для HDFS.

### Базовая идея

Дано: огромный файл. Хотим посчитать что-то по нему.

1. Разделить на части (split).
2. На каждой части запустить функцию **map**, которая выдаёт пары **(ключ, значение)**.
3. Все пары с одинаковым ключом собрать вместе.
4. Для каждого ключа запустить функцию **reduce**, которая сворачивает все значения в итог.

В этом и есть вся магия: огромная задача распилена на много мелких параллельных, плюс одна свёртка.

### Этапы выполнения

```
Input → Split → Map → (Combiner) → Partition → Shuffle & Sort → Reduce → Output
```

**Input / Split.** Входной файл (обычно из HDFS) делится на split-ы. Размер split-а часто совпадает с размером блока HDFS (128 МБ), чтобы каждая задача читала данные с локального диска.

**Map.** Для каждой записи split-а вызывается функция `map(k1, v1)`. Программист реализует её. Возвращает 0..N промежуточных пар `(k2, v2)`. Это первая параллельная стадия.

**Combiner (опционально).** Локальный «мини-reduce» на стороне маппера. Если функция reduce ассоциативна и коммутативна (например, сумма), можно частично свернуть результаты прямо на маппере и не передавать по сети тысячи строк. Это сильно экономит трафик.

**Partitioner.** Определяет, на какой reducer попадёт пара с ключом `k2`. По умолчанию `hash(k2) mod R`, где R — число reducer-ов. Можно написать свой partitioner для специальных распределений.

**Shuffle & Sort.** Самый дорогой этап. Промежуточные пары пересылаются по сети с мапперов на reducer-ы и сортируются по ключу. Если ключей много или они «перекошены», эта стадия становится узким местом.

**Reduce.** Для каждого ключа `k2` вызывается `reduce(k2, [v2, v2, ...])`. Программист реализует свёртку. Результат — пары `(k3, v3)`.

**Output.** Запись итогов в HDFS (или другую файловую систему).

### Классические алгоритмы

**Word Count** — «hello world» MapReduce. Подсчитать частоту каждого слова в гигантском корпусе текстов:

```
map(line):
  for word in split(line):
    emit(word, 1)

reduce(word, counts):
  emit(word, sum(counts))
```

Если документов миллионы, MapReduce параллелит обработку, и каждый узел считает свою часть. На reduce-стадии все «1»-цы для одного слова собираются и складываются.

**Inverted Index** — построение поискового индекса. Для каждого слова сохраняем список документов, в которых оно встречается:

```
map(docId, text):
  for word in text:
    emit(word, docId)

reduce(word, docIds):
  emit(word, unique(docIds))
```

**Группировка и агрегация.** Посчитать сумму заказов по категориям:

```
map(row):
  emit(row.category, row.amount)

reduce(category, amounts):
  emit(category, sum(amounts))
```

**Reduce-side join.** Объединить две таблицы по ключу:

```
map(row_A):
  emit(joinKey, ('A', row_A))

map(row_B):
  emit(joinKey, ('B', row_B))

reduce(key, list):
  # На вход — все строки A и B с этим ключом, делаем декартово произведение
  ...
```

**Map-side join** — если одна из таблиц помещается в память (broadcast), её можно раздать всем мапперам и делать join без shuffle. Это значительно быстрее.

**TF-IDF.** Требует двух-трёх MR-этапов: подсчёт частоты слов в документе, документной частоты, и финальный TF-IDF. Это типичный пример **многоэтапных MR-pipeline**, для которых лучше подходит Spark.

**Сортировка.** Map выдаёт `(значение, _)`. Партиционирование делается по диапазонам, чтобы внутри одного reducer был непрерывный отрезок ключей. Reduce пишет данные упорядоченно.

### Failover и устойчивость

Главный плюс MapReduce — **автоматическая обработка сбоев**.

- Если task падает (узел сгорел, диск сломался) — фреймворк перезапускает его на другом узле.
- При **спекулятивном выполнении** медленные задачи дублируются: запускается копия на другом узле, и принимается тот результат, что пришёл первым.
- Состояние и промежуточные данные хранятся на дисках, что позволяет переживать падения.

### Ограничения и преемники

MapReduce — мощная, но устаревшая модель. Её ограничения:

- **Дорогие I/O между этапами.** Каждый промежуточный результат записывается на диск. Если у нас многоступенчатый пайплайн, мы пишем и читаем терабайты по нескольку раз.
- **Плохо для итеративных алгоритмов.** ML-обучение, графовые алгоритмы, оптимизация — это циклы. На MapReduce они работают медленно.
- **Низкоуровневый API.** Каждая операция — это новый MR-job; сложные пайплайны писать утомительно.

Преемники:

- **Apache Spark** — главная замена. Использует **RDD** и **DataFrame/Dataset**, держит данные в памяти, поддерживает итеративные вычисления. Часто в 10–100 раз быстрее.
- **Apache Flink** — настоящая потоковая обработка с правильной семантикой времени.
- **Apache Beam** — единый API для batch и streaming, может работать поверх Spark, Flink, Google Dataflow.

MapReduce сейчас встречается в основном:
- как фундамент для понимания распределённой обработки;
- в legacy-системах;
- внутри инструментов вроде Apache Hive (хотя Hive давно перешёл на Tez и Spark).

### Что важно сказать в итоге

MapReduce — это модель параллельных вычислений, в которой пользователь пишет две функции — **map** (преобразует записи в промежуточные пары ключ-значение) и **reduce** (сворачивает значения с одинаковым ключом). Фреймворк сам делит данные, планирует задачи, гоняет shuffle и обрабатывает сбои. Классические задачи — word count, инвертированный индекс, агрегации, joins. Главные плюсы — масштабируемость и устойчивость; главные минусы — дорогие I/O и плохое поведение на итеративных алгоритмах. Современные системы (Spark, Flink) развивают идеи MapReduce, давая в десятки раз большую производительность и удобный API.
