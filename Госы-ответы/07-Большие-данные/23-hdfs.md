## Развёрнутый ответ

### Что такое HDFS

**HDFS (Hadoop Distributed File System)** — это распределённая файловая система для хранения очень больших файлов на кластере **commodity-серверов**, то есть обычных недорогих машин. HDFS появилась в проекте Hadoop как открытая реализация архитектуры Google File System (GFS), описанной в 2003 году.

Главная идея — **дёшево хранить большие объёмы и обрабатывать данные параллельно**. Если файл разбит на блоки и каждый блок лежит на нескольких серверах, мы можем читать его одновременно с разных узлов, и при отказе любого диска данные не теряются.

HDFS оптимизирована под несколько специфических сценариев:
- **большие файлы** (от сотен мегабайт до терабайт);
- **потоковое чтение** последовательно, а не случайно;
- **write-once, read-many** — данные пишутся редко, читаются много раз;
- **commodity hardware** — отказы дисков и узлов считаются нормальной частью работы.

### Архитектура

HDFS построена по модели **master-worker**.

```
       Клиент
         │
         ▼
    +----------+
    | NameNode |  ← метаданные, namespace, расположение блоков
    +----------+
       │  │  │
       ▼  ▼  ▼
  +-------+ +-------+ +-------+
  | DN 1  | | DN 2  | | DN 3  |  ← хранят блоки данных
  +-------+ +-------+ +-------+
```

**NameNode** — мастер. Хранит метаданные: дерево файлов и каталогов, права, размеры, **карту блоков** — какой блок какого файла лежит на каких DataNode. Метаданные держатся в оперативной памяти + журналы изменений (**EditLog**) и снимки состояния (**FsImage**) на диске.

**DataNode** — рабочие узлы. Хранят непосредственно блоки данных на локальных дисках. Регулярно (каждые 3 секунды) шлют **heartbeat** NameNode-у, говоря «я жив», и периодически — **block report** со списком блоков, которыми владеют.

**Secondary NameNode** (или в современной конфигурации **Standby NameNode**) сливает EditLog и FsImage, готовя контрольные точки. Это снимает нагрузку с активной ноды.

**HA-режим (High Availability)** — два NameNode: активный и резервный. Журналы хранятся в **JournalNodes**, переключение координируется через **ZooKeeper**. При отказе активного резервный автоматически становится активным за секунды.

### Блоки и репликация

Файл нарезается на **блоки** фиксированного размера. По умолчанию это **128 МБ** (раньше было 64 МБ; для очень больших файлов часто увеличивают до 256 МБ или больше).

Каждый блок **реплицируется** — по умолчанию на 3 DataNode. Это даёт надёжность: при отказе одного или даже двух узлов данные остаются доступны.

Размещение реплик не случайное, а **rack-aware** — учитывает топологию стоек:
- 1-я реплика — на DataNode, ближайшем к писателю;
- 2-я — на DataNode в **другой** стойке;
- 3-я — на ещё одном DataNode в **той же другой** стойке.

Так балансируются риски: даже отказ всей стойки не уничтожает данные.

Если DataNode «пропал» (heartbeat не приходит несколько минут), NameNode инициирует **re-replication** — копирование «осиротевших» блоков на другие узлы, чтобы восстановить нужное число реплик.

### Запись файла

Когда клиент пишет файл, происходит:

1. Клиент обращается к NameNode: «создай файл такой-то».
2. NameNode проверяет права и возвращает список DataNode для первого блока.
3. Клиент устанавливает **pipeline**: DN1 → DN2 → DN3. Шлёт данные первому, тот пересылает второму, второй — третьему.
4. На каждый блок отправляется подтверждение (ack) от хвоста pipeline обратно к клиенту.
5. После завершения блока NameNode даёт список для следующего, и процесс повторяется.

Это даёт хорошую пропускную способность: данные «текут» через цепочку, а не отправляются клиентом во все реплики напрямую.

### Чтение файла

При чтении:

1. Клиент → NameNode: «дай мне список блоков файла X».
2. NameNode возвращает блоки, упорядоченные по близости к клиенту.
3. Клиент читает блоки напрямую с DataNode, минуя NameNode.

Это и есть **data locality**: задание обработки запускается там, где лежит блок, чтобы не гонять данные по сети. Hadoop YARN и Spark об этом знают и стараются назначать задачи близко к данным.

### Особенности

**Write-once, read-many.** Файл пишется один раз и потом только читается. Запись только в конец (append) допустима, но случайных обновлений по середине файла HDFS не поддерживает. Это серьёзное ограничение, но оно упрощает архитектуру и подходит под аналитические нагрузки.

**Большие файлы.** HDFS неэффективна для миллионов маленьких файлов — каждый файл создаёт запись в памяти NameNode. Если файлов много и они мелкие, NameNode становится узким местом. Это известная проблема, для которой существуют workaround: HAR-архивы, Avro/Parquet с группировкой мелких записей.

**Snapshots.** HDFS поддерживает мгновенные снимки директорий — удобно для бэкапов.

**Erasure Coding (с Hadoop 3.0).** Альтернатива репликации, экономит место. Вместо 3 копий блока (overhead 200%) используется код Рида-Соломона — например, RS(10,4): 10 блоков данных + 4 блока чётности, overhead 40%. Минус — выше расход CPU при чтении после отказов.

**Квоты.** Можно ограничивать дисковое пространство и число объектов на пользователя или каталог.

### Команды HDFS

Базовая работа из командной строки:

```bash
hdfs dfs -ls /user/matvey
hdfs dfs -mkdir /data
hdfs dfs -put local.txt /data/
hdfs dfs -get /data/local.txt .
hdfs dfs -cat /data/local.txt | head
hdfs dfs -du -h /data
hdfs dfs -rm -r /data/old
hdfs fsck /data/file.parquet -files -blocks -locations
hdfs dfsadmin -report
```

Команда `fsck` показывает целостность файлов, `dfsadmin -report` — состояние кластера.

### Современный контекст

HDFS остаётся стандартом on-premise Hadoop-кластеров, но в облачную эру её активно заменяет **S3-совместимое объектное хранилище** (AWS S3, Google Cloud Storage, MinIO, Ceph) в связке с табличными форматами **Iceberg / Delta Lake / Hudi**. Это даёт:
- независимое масштабирование хранилища и вычислений;
- более низкую стоимость;
- удобную работу из любых движков (Spark, Trino, Flink, ClickHouse);
- ACID-транзакции и time travel за счёт табличного формата.

**Apache Ozone** — преемник HDFS от той же команды, объектное хранилище с архитектурой, лишённой узких мест NameNode.

HDFS отлично сочетается со Spark, Hive, Impala, MapReduce, HBase. Для новых проектов в облаке обычно выбирают S3 + Iceberg; для крупных on-prem-кластеров HDFS — всё ещё разумный выбор.

### Что важно сказать в итоге

HDFS — это распределённая файловая система Hadoop, рассчитанная на хранение больших файлов на кластере дешёвых серверов с автоматической репликацией и отказоустойчивостью. Архитектура «master-worker»: NameNode хранит метаданные, DataNode — блоки данных. Главные принципы — большие блоки (128 МБ), репликация (×3), data locality, write-once read-many. HDFS оптимальна для аналитических нагрузок и плохо подходит для миллионов мелких файлов и случайных обновлений. Современный тренд — переход на S3-совместимое хранилище + Iceberg/Delta для cloud-native lakehouse-архитектуры, но HDFS остаётся стандартом для on-prem Hadoop.
