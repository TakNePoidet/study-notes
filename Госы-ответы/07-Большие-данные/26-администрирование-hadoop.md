## Развёрнутый ответ

### Состав Hadoop-кластера

Кластер Hadoop — это связка из нескольких сервисов:

- **HDFS** — распределённая файловая система (NameNode + DataNode + опц. JournalNode).
- **YARN** — менеджер ресурсов (ResourceManager + NodeManager).
- **Сервисы экосистемы** — Hive Metastore, HiveServer2, HBase, Oozie, Zeppelin, Hue, Spark on YARN, Tez.
- **ZooKeeper** — координация HA, выборы лидера.

Типичная роль узлов:

| Компонент | Роль |
|-----------|------|
| **NameNode** | Метаданные HDFS, namespace |
| **DataNode** | Хранят блоки данных |
| **JournalNode** | Журналы для HA NameNode |
| **ResourceManager (YARN)** | Управляет ресурсами кластера |
| **NodeManager (YARN)** | Запускает контейнеры на узле |
| **ApplicationMaster** | Координирует конкретное приложение |
| **ZooKeeper** | HA-координация |

### Установка и конфигурация

Есть два подхода: **ручная установка** или через дистрибутив с графическим менеджером.

**Дистрибутивы:**
- **Cloudera CDP** — коммерческий, проприетарный;
- **Apache Hadoop** — open-source, требует ручной настройки;
- **Arenadata Hadoop (ADH)** — российский, на базе Apache, с поддержкой и инструментами управления.

**Менеджеры:**
- **Apache Ambari** — был open-source менеджером, сейчас в архиве;
- **Cloudera Manager** — основной для CDP;
- **Arenadata Cluster Manager** — для ADH.

**Главные конфиги:**

- `core-site.xml` — общие настройки.
- `hdfs-site.xml` — настройки HDFS (размер блока, replication factor).
- `yarn-site.xml` — настройки YARN.
- `mapred-site.xml` — настройки MapReduce.
- `capacity-scheduler.xml` — настройки очередей.

Важные параметры: размер блока (128/256 МБ), фактор репликации (3), пути монтирования дисков (JBOD предпочтительнее RAID для DataNode), heap-памяти JVM для NameNode (критично при большом namespace).

### YARN — управление ресурсами

**YARN** разделяет кластер на ресурсы (память + vCores) и распределяет их между приложениями.

Каждое приложение (Spark job, MR job) при запуске:
1. ResourceManager создаёт **ApplicationMaster** для job.
2. AM запрашивает у RM **контейнеры** с нужным объёмом памяти и vCores.
3. RM назначает контейнеры на NodeManager-ы.
4. NodeManager-ы запускают исполнители задачи.

Это даёт изоляцию и контроль ресурсов.

**Планировщики:**

- **Capacity Scheduler** — деление кластера на **очереди** с гарантированной и максимальной долей ресурсов. Подходит для multi-tenant сред.
- **Fair Scheduler** — задачи получают ресурсы пропорционально, без жёстких квот.

**Queue ACL** — кто может запускать в какую очередь.

### Безопасность

В production-кластере обязательны:

**Kerberos** — главный механизм аутентификации. Все сервисы используют принципалы и keytab-файлы; пользователи получают тикеты через `kinit`. Без Kerberos кластер открыт.

**Apache Ranger** — централизованные политики авторизации для HDFS, Hive, HBase, Kafka, YARN. Удобный UI для управления правами.

**Apache Knox** — единая точка входа в кластер: проксирует REST-API сервисов, добавляет SSO.

**Шифрование в покое:**
- **HDFS Transparent Data Encryption (TDE)** с **KMS (Key Management Server)** — шифрование на уровне зон HDFS.
- Шифрование дисков на уровне ОС.

**Шифрование в канале** — TLS для всех RPC, UI и веб-интерфейсов.

**Аудит** — журналы Ranger, NameNode audit log, YARN audit log.

### Мониторинг и эксплуатация

**Web UI** компонентов:

- HDFS Web UI на порту 9870 (раньше 50070).
- YARN ResourceManager UI на 8088.
- Job History Server для завершённых MR/Spark job.

**Метрики:**

- JMX-метрики от всех JVM-сервисов;
- экспорт в Prometheus через **JMX Exporter**;
- готовые дашборды в Grafana.

**Логи:**

- централизованный сбор (Filebeat → Elasticsearch / Vector → Loki);
- `yarn logs -applicationId application_xxx` — логи приложения после завершения.

**Команды администратора:**

```bash
hdfs dfsadmin -report          # состояние кластера HDFS
hdfs fsck /                    # целостность файлов
yarn node -list                # список нод YARN
yarn application -list         # запущенные приложения
yarn application -kill app_id  # убить приложение
```

### Балансировка и обслуживание

**HDFS Balancer.** Если на одних DataNode заполнено 90%, а на других — 30%, балансер перераспределит блоки. Запускается командой `hdfs balancer`.

**Disk Balancer.** Балансировка внутри одной DataNode по разным дискам.

**Decommission / Recommission.** Плавный вывод узла из кластера: его блоки реплицируются на другие узлы, после чего он отключается. Обратная процедура — добавление.

**Hypercare.** Регулярная очистка:
- `.Trash` пользователей (по умолчанию 7 дней);
- старые job-логи и истории;
- неиспользуемые snapshots.

**Обновления.** Rolling upgrade — обновление сервисов по очереди без остановки кластера. Для NameNode требуется HA.

### Резервное копирование и восстановление

- **Snapshots HDFS** — мгновенные срезы директорий.
- **DistCp** — параллельное копирование между кластерами, основной инструмент DR (disaster recovery).
- **Бэкап метаданных NameNode** (FsImage + EditLog) — критично, без них кластер не восстановить.
- **Бэкап Hive Metastore** — обычная БД (PostgreSQL/MySQL), бэкапится стандартными средствами.

### Производительность и проблемы

Типовые проблемы и решения:

- **Высокое использование heap NameNode** при множестве мелких файлов — нужно использовать **compaction**: объединять мелкие файлы в крупные Parquet/Avro.
- **Slow shuffle в MR/Spark** — настроить сжатие промежуточных данных, увеличить число reducer-ов.
- **Skew** — пере-партиционирование, salting.
- **Сетевая топология** — корректная **rack awareness** в конфиге.

### Связанные сервисы экосистемы

- **Hive** — SQL-движок, Metastore + HiveServer2.
- **HBase** — NoSQL колоночная БД поверх HDFS.
- **Spark on YARN** — основная нагрузка сегодня.
- **Oozie** / **Airflow** — оркестрация workflow.
- **Kafka** — стриминг.
- **NiFi** — пайплайны интеграции.

### Современный контекст

В 2026 году ландшафт меняется. Многие компании уходят от классического Hadoop on-prem в сторону:

- **Spark on Kubernetes** + **S3 + Iceberg** — современный cloud-native стек;
- **Managed-сервисы** — Yandex DataProc, AWS EMR, Databricks;
- **Apache Ozone** как замена HDFS.

Hadoop сейчас — это либо legacy в крупных корпорациях, либо изолированные сценарии (госорганизации, КИИ, военка), где облако недопустимо. Понимание администрирования Hadoop остаётся востребованным, но новые проекты обычно стартуют без него.

### Что важно сказать в итоге

Hadoop-кластер — это HDFS (NameNode + DataNode) + YARN (ResourceManager + NodeManager) + сервисы экосистемы. Администрирование включает установку (дистрибутивы CDP, ADH), конфигурацию, управление ресурсами через YARN, безопасность (Kerberos, Ranger, Knox, шифрование), мониторинг (JMX, Prometheus, UI), балансировку, бэкапы и обновления. Современный тренд — переход от классического Hadoop к cloud-native lakehouse-стекам (Spark + S3 + Iceberg), но понимание администрирования Hadoop остаётся важным для legacy и закрытых сред.
