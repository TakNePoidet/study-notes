# 37. Проектирование программного обеспечения: разработка спецификации требований, модель вариантов использования. Архитектурный анализ. Модель классов анализа. Модель взаимодействия

[← К группе «Проектирование ПО»](README.md) · [← Ко всем группам](../README.md)

## План ответа

1. Виды требований и как их формулировать.
2. Что такое use case и как его документируют.
3. Архитектурный анализ как переход от требований к решению.
4. Модель классов анализа: Boundary / Control / Entity.
5. Модель взаимодействия: диаграммы последовательностей и коммуникаций.

## Развёрнутый ответ

### Виды требований

При проектировании ПО все требования делятся на несколько классов.

**Функциональные** — что система должна делать. «Пользователь может войти по email и паролю», «администратор видит список заявок», «система отправляет SMS при оплате».

**Нефункциональные (NFR)** — как именно система это делает: время отклика, доступность (SLA), безопасность, удобство, масштабируемость, переносимость, локализация, поддерживаемость.

**Бизнес-требования** — цели заказчика на более высоком уровне: «увеличить конверсию на 15%», «сократить время оформления заказа в 2 раза».

**Ограничения** — рамки, которые нельзя нарушить: технологический стек, регуляторика (152-ФЗ, GDPR), бюджет, сроки.

Различение этих классов помогает не упустить важное. Часто проекты «спотыкаются» именно об NFR: систему сделали функционально правильной, но она тормозит, не масштабируется или небезопасна.

### Источники требований

Требования собирают через:

- **интервью** с заказчиком, стейкхолдерами, конечными пользователями;
- **наблюдение** за тем, как люди сейчас работают (особенно ценно для AS-IS);
- **мозговые штурмы и воркшопы**;
- **прототипирование** и обсуждение макетов;
- **анализ конкурентов и аналогов**;
- **данные аналитики** (если есть существующий продукт);
- **отзывы пользователей**, обращения в поддержку.

Качественный сбор требований — это половина успеха проекта.

### Хорошее требование

Критерии хорошего требования формулируются через несколько эвристик:

**SMART** — Specific (конкретно), Measurable (измеримо), Achievable (достижимо), Relevant (релевантно), Time-bound (привязано ко времени).

**INVEST** (для user stories) — Independent (независимо), Negotiable (обсуждаемо), Valuable (ценно), Estimable (оценимо), Small (мало), Testable (тестируемо).

Шаблон user story:

> Как **[роль]**, я хочу **[действие]**, чтобы **[ценность]**.
> Acceptance Criteria: 1) … 2) … 3) …

User story не описывает реализацию; она описывает **потребность** и условия приёмки.

### Use case (сценарий использования)

**Use case** — это описание сценария взаимодействия акторов с системой. Use case задаёт функциональные требования через истории. Это более структурированный, чем user story, формат — лучше подходит для сложных систем.

Шаблон:

```
ID: UC-12
Название: Войти в систему
Актор: Пользователь
Предусловие: Аккаунт зарегистрирован и не заблокирован.

Основной поток:
  1. Пользователь открывает страницу логина.
  2. Вводит email и пароль.
  3. Нажимает «Войти».
  4. Система проверяет учётные данные.
  5. Система создаёт сессию и открывает личный кабинет.

Альтернативные потоки:
  4a. Неверный пароль — вывести ошибку, дать 5 попыток за 15 минут.
  4b. Аккаунт заблокирован — предложить восстановление через email.

Постусловие: Создана активная сессия пользователя.
```

Use case-диаграмма UML показывает акторов и связанные use case, плюс стереотипы `<<include>>` (общий поведенческий блок включается) и `<<extend>>` (опциональное расширение). На диаграмме хорошо видно функциональное «лицо» системы — то, что она делает с точки зрения внешнего мира.

### Архитектурный анализ

После того как требования собраны, начинается **переход от «что» к "как"**. Это и есть архитектурный анализ — выработка решения, обеспечивающего выполнение требований.

Что делается на этом этапе:

- **Разделение на слои** — типичные паттерны Presentation / Application / Domain / Infrastructure, или Hexagonal / Clean Architecture.
- **Выбор архитектурного стиля** — монолит, модульный монолит, микросервисы, event-driven, serverless.
- **Идентификация bounded contexts** — границ доменов (DDD).
- **Технологические решения** — какая БД, какой кэш, какие очереди, какие фреймворки.
- **Документирование решений** в формате ADR (Architecture Decision Records).

Хорошо проведённый архитектурный анализ закладывает основу качества системы на годы вперёд.

### Модель классов анализа

В RUP-подходе для проектирования вводится **трёхчастная классификация классов**, помогающая разделить обязанности и не сваливать всё в одну кучу.

**Entity (сущности)** — доменные классы, моделирующие предметную область: `User`, `Order`, `Product`, `Invoice`. Содержат бизнес-данные и поведение.

**Boundary (границы)** — взаимодействие с внешним миром: формы UI, контроллеры API, представления страниц. Они принимают входы и отдают результат.

**Control (управление)** — оркестраторы сценария use case: координируют entity и boundary, реализуют логику процесса. Например, `LoginController`, `OrderService`, `PaymentProcessor`.

Эта модель упрощает мышление: ясно, какой класс отвечает за что. Например, для UC «Войти»:

```
LoginForm (Boundary) ──── LoginController (Control) ──── User (Entity)
                                       │
                                       └── SessionService (Control)
```

Этот подход хорошо ложится на современные слоистые архитектуры: Entity = доменный слой, Control = сервисный слой, Boundary = представительский слой.

### Модель взаимодействия

После того как мы выделили классы, нужно понять, **как они общаются между собой**. Для этого в UML есть **диаграммы взаимодействия**: последовательностей и коммуникаций.

**Sequence Diagram (диаграмма последовательностей)** — наиболее распространённая. Показывает, какие сообщения и в каком порядке отправляются между объектами по времени. Время идёт сверху вниз.

Пример для UC «Войти»:

```
Actor "Пользователь" -> LoginForm: Ввести email, пароль
LoginForm -> LoginController: submit(email, pw)
LoginController -> UserRepository: findByEmail(email)
UserRepository --> LoginController: User
LoginController -> User: verifyPassword(pw)
User --> LoginController: true
LoginController -> SessionService: create(userId)
SessionService --> LoginController: token
LoginController --> LoginForm: redirect(/dashboard)
```

Эта диаграмма понятна без дополнительных пояснений: видно, кто кому что вызывает, какие данные текут, где может «застрять» сценарий.

**Communication Diagram (диаграмма коммуникаций)** показывает те же сообщения, но с акцентом на **структурных связях**: какие объекты с какими общаются. Менее детальная по времени, но лучше отображает топологию.

### Управление требованиями

В реальном проекте требования живут долго и меняются. Важно:

- **Версионирование** требований — в Jira, Confluence, IBM DOORS, ReqIF.
- **Трассируемость (traceability)** — каждое требование связано с источником (откуда взялось), артефактами (use case, классы) и тестами (как проверяется).
- **Регулярные ревью** со стейкхолдерами.
- **Прототипирование** для быстрой обратной связи.
- **DoR / DoD** — Definition of Ready и Definition of Done для каждой задачи.

### Что важно сказать в итоге

Спецификация требований начинается с разделения на функциональные, нефункциональные, бизнес-требования и ограничения. Use case описывают функциональность через сценарии взаимодействия акторов с системой. Архитектурный анализ переводит требования в решение — слои, технологии, границы доменов. Модель классов анализа делит классы на Entity, Boundary и Control, разделяя ответственность. Модель взаимодействия (sequence/communication diagrams) показывает, как эти классы общаются для реализации use case. Хорошее проектирование — это сквозная связь от бизнес-требования до конкретного класса и теста.
