# 34. Организация проектирования программного обеспечения: процессы жизненного цикла программного обеспечения, модели проектирования и разработки программного обеспечения

[← К группе «Проектирование ПО»](README.md) · [← Ко всем группам](../README.md)

## План ответа

1. Что такое жизненный цикл ПО и зачем его описывать.
2. Стандарт ISO/IEC/IEEE 12207 и группы процессов.
3. Типовые стадии: анализ → требования → проектирование → реализация → тестирование → внедрение → сопровождение.
4. Модели разработки: водопад, V, итеративная, спиральная, RUP, Agile, DevOps.
5. Как выбирать модель под проект.

## Развёрнутый ответ

### Что такое жизненный цикл ПО

**Жизненный цикл программного обеспечения (Software Development Life Cycle, SDLC)** — это последовательность стадий, которые проходит программный продукт от идеи до вывода из эксплуатации. Жизненный цикл описывает, **что и в каком порядке делается**, какие артефакты создаются и какие роли участвуют.

Понятие жизненного цикла полезно сразу по нескольким причинам:

- даёт **единый язык** для команды и заказчика;
- помогает **планировать** работы и ресурсы;
- позволяет применять **стандарты качества** и **регуляторные требования**;
- упрощает **аудит и сопровождение** долгоживущих систем.

### Стандарт ISO/IEC/IEEE 12207

Главный международный стандарт, описывающий процессы жизненного цикла ПО, — **ISO/IEC/IEEE 12207** (актуальная редакция — 2017 г.). Он группирует процессы в три категории.

**Основные процессы** — без них продукта не будет:

- приобретение (acquisition);
- поставка (supply);
- разработка (development);
- эксплуатация (operation);
- сопровождение (maintenance);
- вывод из эксплуатации (disposal).

**Вспомогательные процессы** — обеспечивают качество и контроль:

- документирование;
- управление конфигурациями;
- обеспечение качества (QA);
- верификация и валидация;
- совместный анализ, аудит;
- решение проблем.

**Организационные процессы** — относятся ко всей организации:

- управление (management);
- инфраструктура;
- улучшение;
- обучение.

### Типовые стадии разработки

В большинстве моделей в том или ином виде встречаются одни и те же стадии.

**1. Анализ потребностей.** Понять, какую бизнес-проблему решаем, кому продукт нужен, как он впишется в существующие процессы. На этом этапе работают аналитики и продакт-менеджеры.

**2. Сбор требований.** Функциональные требования (что делает система) и нефункциональные (как — скорость, безопасность, доступность). Артефакты: SRS, user stories, use cases.

**3. Проектирование.** Архитектура системы, дизайн интерфейсов, выбор технологий, проектирование данных. Создаются ADR, диаграммы, прототипы.

**4. Реализация (кодирование).** Разработчики пишут код согласно проектным решениям. Параллельно — code review, статический анализ, юнит-тесты.

**5. Тестирование.** Юнит-, интеграционное, системное, приёмочное (UAT). Может включать нагрузочное, тестирование безопасности, юзабилити.

**6. Внедрение (deployment).** Развёртывание в продуктивной среде, миграция данных, обучение пользователей.

**7. Сопровождение.** Поддержка, исправление ошибок, мелкие улучшения, развитие функциональности. Обычно длится значительно дольше, чем разработка.

**8. Вывод из эксплуатации.** Архивирование данных, миграция на новые системы, окончание поддержки.

### Модели разработки

**Водопад (Waterfall).** Классическая линейная модель. Каждая стадия начинается после полного завершения предыдущей, между ними — формальное согласование. Плюсы: предсказуемость, обширная документация, ясные роли. Минусы: не выдерживает изменений требований, дефекты ловятся слишком поздно. Подходит для проектов с фиксированными требованиями и регуляторикой (медицина, авиация, гособоронзаказ).

**V-модель.** Расширение водопада, где каждой стадии слева соответствует уровень тестирования справа: требования ↔ приёмочные тесты, архитектура ↔ интеграционные тесты, дизайн ↔ системные тесты, реализация ↔ юнит-тесты. Это даёт более сильное внимание к качеству на каждом уровне.

**Итеративная модель.** Продукт строится итерациями, каждая улучшает функциональность. Раньше получаем обратную связь, можем менять курс.

**Инкрементная модель.** Похожа на итеративную, но каждая итерация добавляет новый функциональный «кусок».

**Спиральная модель (Боэм, 1988).** Каждый «виток» спирали проходит четыре фазы: определение целей → анализ рисков → разработка и проверка → планирование следующего витка. Хороша для крупных проектов с большими рисками: каждая итерация прямо адресует самые опасные неизвестные.

**RUP (Rational Unified Process).** Итеративная и use-case-ориентированная методология IBM. Делит проект на четыре фазы: **Inception** (определение видения), **Elaboration** (проработка архитектуры и ключевых рисков), **Construction** (основной объём разработки), **Transition** (передача пользователям). Внутри каждой фазы может быть несколько итераций.

**Agile / Scrum / Kanban.** Семейство гибких методологий — короткие итерации, частая поставка, кросс-функциональные команды, готовность к изменениям. Самые популярные модели сегодня для продуктовой разработки.

**DevOps / CI-CD.** Не отдельная модель, а культура и набор практик, ускоряющих цикл «разработка → эксплуатация». Включает: **Continuous Integration** (автоматическая интеграция кода с тестами), **Continuous Delivery / Deployment** (автоматическая выкладка), **Infrastructure as Code**, мониторинг, обсервабилити, SRE-подход.

### Как выбирать модель

Выбор модели зависит от характера проекта.

| Условия | Подходящая модель |
|---------|-------------------|
| Требования стабильны, регуляторика | Водопад, V-модель |
| Высокие технические риски | Спиральная, RUP |
| Часто меняющиеся требования | Agile, Scrum, Kanban |
| Стартап, поиск product-market fit | Lean Startup + Agile |
| Зрелый продукт, частые релизы | DevOps + Kanban |
| Крупная enterprise-разработка | SAFe, LeSS — масштабированные Agile |

На практике часто встречается **гибрид**: верхний уровень управления — водопадный (для контракта, milestone), а внутри команды — Scrum или Kanban. Это называется hybrid PM.

### Хорошие практики на любом цикле

Независимо от модели, есть набор практик, делающих жизненный цикл здоровым:

- **управление версиями кода** (Git, trunk-based development);
- **автотесты** как часть жизненного цикла — юнит-, интеграционные, e2e;
- **code review** и парное программирование;
- **документирование решений** через ADR;
- **CI/CD-пайплайны** с автоматической проверкой качества;
- **мониторинг и SLO** в эксплуатации;
- **постмортемы** по инцидентам без поиска виноватых.

### Что важно сказать в итоге

Жизненный цикл ПО — это структурированное описание стадий от идеи до вывода продукта из эксплуатации. Стандарт ISO/IEC/IEEE 12207 описывает основные, вспомогательные и организационные процессы. Существуют разные модели разработки: водопад и V для стабильных требований, итеративная и спиральная для рисковых проектов, RUP как зрелая итеративная методология, Agile-семейство для продуктовой разработки, DevOps как культура ускорения цикла. Выбор модели — это компромисс между предсказуемостью и гибкостью; современный тренд — гибридные подходы, сочетающие достоинства разных моделей.
