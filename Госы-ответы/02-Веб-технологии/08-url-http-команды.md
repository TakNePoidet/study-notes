## Развёрнутый ответ

### Система адресации URL

**URL (Uniform Resource Locator)** — это строка, которая однозначно адресует ресурс в интернете и одновременно описывает, как до него добраться. URL — частный случай более общего понятия **URI (Uniform Resource Identifier)**.

Структура URL описана в RFC 3986. Полный формат:

```
scheme://user:password@host:port/path?query#fragment
```

Пример: `https://user:pass@example.com:443/blog/post?id=42&sort=asc#comments`.

Разберём по частям:

- **scheme** — протокол (`https`, `http`, `ftp`, `mailto`, `file`, `ws`, `wss`). Определяет, как обращаться к ресурсу.
- **user:password** — учётные данные (исторически, сейчас редко).
- **host** — доменное имя или IP-адрес сервера.
- **port** — порт. Если опущен, берётся стандартный для схемы: 80 для http, 443 для https.
- **path** — путь к ресурсу на сервере, аналог пути в файловой системе.
- **query** — параметры запроса после `?`, в формате `key=value`, разделённых амперсандом.
- **fragment** — часть после `#`, идентифицирует фрагмент внутри ресурса (якорь на странице) и **не отправляется на сервер**.

Понимание структуры URL важно, потому что от него зависит и SEO, и аналитика, и безопасность (например, защита от open redirect).

### Сетевые диагностические команды

В арсенале любого айтишника есть набор команд для диагностики сети.

**ping** — отправляет ICMP Echo Request на указанный узел и ждёт Echo Reply. По времени отклика (RTT) и проценту потерь можно оценить, доступен ли узел и в каком состоянии канал. Минусы: некоторые сети блокируют ICMP, и тогда ping не работает, хотя сам узел доступен по другим протоколам.

```bash
ping -c 4 8.8.8.8
ping example.com
```

**tracert (Windows) / traceroute (Linux/Mac)** — показывает маршрут пакетов до удалённого узла. Работает хитро: отправляет пакеты с возрастающим значением TTL. Когда TTL обнуляется, маршрутизатор возвращает ICMP «Time Exceeded», и мы узнаём его адрес. Так шаг за шагом строится цепочка хопов.

```bash
tracert example.com           # Windows
traceroute -n example.com     # Linux/Mac, -n чтобы не резолвить DNS
```

**netstat** (или современная замена `ss` в Linux) — показывает состояние сетевых соединений и слушающих портов. Используется, чтобы понять, какой процесс слушает на каком порту, какие соединения установлены, сколько находится в состояниях TIME_WAIT, ESTABLISHED, LISTEN.

```bash
netstat -tulnp      # старая, Linux
ss -tulnp           # современная замена
netstat -an         # Windows
```

Эти команды — базовый инструмент для диагностики проблем сети: «сайт не открывается», «приложение не подключается», «потери на каком-то участке».

### Стандартизация веба

Интернет и веб развиваются благодаря согласованным стандартам, которые публикуют несколько международных организаций.

**ISOC (Internet Society)** — материнская некоммерческая организация, основанная в 1992 году. Она координирует развитие интернета и поддерживает работу нижестоящих структур, прежде всего IETF.

**IETF (Internet Engineering Task Force)** — рабочая группа, разрабатывающая технические стандарты протоколов интернета. Документы IETF называются **RFC (Request for Comments)**. Каждый RFC имеет номер и описывает определённый стандарт или практику. Например, HTTP/1.1 описан в RFC 9110/9112 (актуальная версия), TCP — в RFC 9293, IP — в RFC 791.

**W3C (World Wide Web Consortium)** — организация, разрабатывающая стандарты веба: HTML, CSS, DOM, WebAuthn, ARIA. Основана Тимом Бернерсом-Ли в 1994 году.

**WHATWG (Web Hypertext Application Technology Working Group)** — образовалась в 2004-м как реакция на медленное развитие HTML в W3C. Сейчас именно WHATWG поддерживает «Living Standard» для HTML, DOM и Fetch — это фактический источник истины для современных браузеров.

Эта экосистема обеспечивает совместимость: любой браузер, любой веб-сервер, любой клиент во всём мире могут общаться, потому что говорят на одном языке стандартов.

### HTTP — протокол передачи гипертекста

**HTTP (HyperText Transfer Protocol)** — это прикладной протокол передачи данных, лежащий в основе веба. По модели «запрос — ответ»: клиент шлёт запрос, сервер возвращает ответ. Изначально текстовый, поверх TCP.

Базовая структура запроса:

```http
GET /api/users/42 HTTP/1.1
Host: example.com
Accept: application/json
Authorization: Bearer eyJhbGciOi...
```

Ответ:

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Cache-Control: max-age=60

{"id":42,"name":"Matvey"}
```

В HTTP различают **методы** (verbs): `GET` (чтение), `POST` (создание/действие), `PUT` (полная замена), `PATCH` (частичное обновление), `DELETE` (удаление), `HEAD` (только заголовки), `OPTIONS` (узнать поддерживаемые методы, используется в CORS).

Коды ответа сгруппированы по сотням:

- **1xx** — информационные (редко используются явно).
- **2xx** — успех: 200 OK, 201 Created, 204 No Content.
- **3xx** — перенаправление: 301 Moved Permanently, 302 Found, 304 Not Modified.
- **4xx** — ошибка клиента: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests.
- **5xx** — ошибка сервера: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable.

Заголовки переносят метаданные: тип содержимого, кэширование, аутентификация, длина, кодировка, информация о клиенте.

### Версии HTTP

**HTTP/1.0** (1996) — каждое соединение для одного запроса, что было крайне неэффективно.

**HTTP/1.1** (1997, обновлён в RFC 9110/9112) — добавил **keep-alive**: одно TCP-соединение для нескольких запросов. Это основная версия в эпоху раннего веба.

**HTTP/2** (2015) — бинарный протокол поверх одного TCP-соединения с мультиплексированием: несколько запросов и ответов идут параллельно без блокирующего ожидания. Сжатие заголовков (HPACK), серверный push (сейчас deprecated). HTTP/2 значительно ускорил загрузку страниц.

**HTTP/3** (2022, RFC 9114) — построен поверх **QUIC** (UDP + TLS 1.3 + мультиплексирование). Решает проблему head-of-line blocking, существовавшую в HTTP/2 на уровне TCP. Лучше работает в сетях с потерями и при переключении между Wi-Fi и LTE.

Параллельно с HTTP в современном вебе живут **WebSocket** (двусторонний поток поверх HTTP-upgrade), **Server-Sent Events** (стрим от сервера к клиенту) и **gRPC** (бинарный RPC поверх HTTP/2).

### HTTPS

HTTPS — это тот же HTTP, но поверх **TLS**. TLS обеспечивает шифрование канала, аутентификацию сервера (через сертификаты от CA — Certificate Authority) и целостность данных. Сертификаты сегодня в массе бесплатны благодаря Let’s Encrypt, и весь современный веб ходит по HTTPS. HTTP без шифрования браузеры открыто помечают как небезопасный.

### Что важно сказать в итоге

URL — это структурированный адрес ресурса со схемой, хостом, путём и параметрами. Базовый инструмент диагностики сети — ping, traceroute и netstat. Веб-стандарты разрабатывает целая экосистема: ISOC и IETF выпускают RFC по протоколам, W3C и WHATWG — стандарты HTML/CSS/DOM. HTTP — прикладной протокол запрос/ответ, прошедший путь от текстового HTTP/1.1 через бинарный HTTP/2 к HTTP/3 поверх QUIC; современный HTTPS добавляет шифрование TLS и стал стандартом де-факто.
