# 7. Основы Веб-технологий: DNS — система доменных имён. Технология «клиент-сервер». Web-серверы, прокси-серверы

[← К группе «Веб-технологии»](README.md) · [← Ко всем группам](../README.md)

## План ответа

1. Что такое DNS, зачем он нужен, иерархическая структура и типы записей.
2. Как происходит разрешение доменного имени в IP-адрес.
3. Клиент-серверная архитектура: что это, чем отличается от P2P.
4. Что такое web-сервер, его задачи и популярные представители.
5. Прокси-сервер: прямые (forward) и обратные (reverse) прокси.

## Развёрнутый ответ

### Что такое DNS

**DNS (Domain Name System)** — это иерархическая распределённая система, которая преобразует доменные имена (например, `example.com`) в IP-адреса (например, `93.184.216.34`) и обратно. Без DNS пришлось бы запоминать числовые адреса всех сайтов и серверов, что для современного интернета невозможно.

DNS появился в 1983 году как замена единого файла `hosts.txt`, который раньше распространялся между всеми компьютерами сети. По мере роста интернета такой подход стал неработоспособным, и Пол Мокапетрис предложил распределённую систему, где разные части пространства имён обслуживают разные серверы.

### Иерархия DNS

DNS устроен как дерево. На самом верху — **корневая зона** (`.`), которой управляют 13 наборов корневых серверов по всему миру. Под корневой зоной идут **TLD (Top-Level Domains)**: `.com`, `.org`, `.ru`, `.io` и сотни других. Каждый TLD управляется своим регистратором (например, `.ru` — Координационным центром доменов `.RU/.РФ`).

Под TLD находятся домены второго уровня — `example.com`, `yandex.ru` — которыми владеют конкретные организации. Далее идут поддомены произвольной глубины: `www.example.com`, `mail.example.com`, `api.v2.example.com`.

Каждая зона хранится на **авторитативных серверах**, которые знают «правду» о данной зоне. Дополнительно работают **рекурсивные резолверы** — серверы (обычно DNS-провайдера, Google 8.8.8.8, Cloudflare 1.1.1.1), которые от имени клиента опрашивают цепочку авторитативных серверов и кэшируют результат.

### Типы DNS-записей

В DNS хранятся не только адреса. Основные типы записей:

- **A** — IPv4-адрес для домена.
- **AAAA** — IPv6-адрес.
- **CNAME** — псевдоним, ссылка на другое имя.
- **MX** — почтовые серверы для домена.
- **TXT** — произвольный текст (используется для SPF, DKIM, верификации владельцев).
- **NS** — какие серверы являются авторитативными для зоны.
- **PTR** — обратное преобразование IP в имя.
- **SOA** — метаданные зоны (Start of Authority).

### Как происходит разрешение имени

Когда пользователь набирает `example.com` в браузере, происходит следующее:

1. Браузер сначала смотрит в **локальный кэш** — операционная система и сам браузер кэшируют недавние ответы.
2. Если в кэше нет — запрос уходит к **рекурсивному резолверу**, обычно настроенному в системе (DNS провайдера или публичный, например `8.8.8.8`).
3. Резолвер опрашивает корневые серверы и узнаёт, кто отвечает за `.com`.
4. Идёт к серверам `.com` и узнаёт, кто отвечает за `example.com`.
5. Идёт к авторитативным серверам `example.com` и получает A-запись.
6. Возвращает ответ клиенту и кэширует его на время, указанное в TTL записи.

Это «дорогой» путь, поэтому DNS активно использует кэширование на всех уровнях.

### Технология «клиент-сервер»

**Клиент-серверная архитектура** — это модель взаимодействия, в которой одна сторона (клиент) запрашивает услугу или данные, а другая (сервер) предоставляет их. Связь идёт по сети, чаще всего по протоколу TCP/IP.

Клиент обычно инициирует запрос и ждёт ответ. Это может быть «тонкий клиент» — например, браузер, который сам по себе мало что умеет, и почти всё происходит на сервере. Или «толстый клиент» — например, десктоп-приложение с большим объёмом локальной логики, обращающееся к серверу только за специфическими данными.

Сервер — это программа, обычно работающая постоянно и обслуживающая много клиентов одновременно. Он хранит общие ресурсы, обеспечивает безопасность, согласованность, разграничение доступа.

Альтернативная архитектура — **peer-to-peer (P2P)**, где все участники равноправны: каждый одновременно и клиент, и сервер. BitTorrent, IPFS, блокчейн-сети — типичные примеры.

В вебе обычно встречается **трёхзвенная архитектура**: клиент (браузер) ↔ сервер приложений ↔ база данных. Это разделение на слои позволяет масштабировать каждый независимо. В более крупных системах используется **N-tier** или **микросервисная** архитектура: множество специализированных сервисов общаются между собой.

### Web-сервер

**Web-сервер** — это программа, которая принимает HTTP/HTTPS-запросы и возвращает ответы: HTML-страницы, картинки, CSS, JS, JSON-ответы API. Web-сервер слушает TCP-порт (обычно 80 для HTTP, 443 для HTTPS), разбирает входящий запрос, генерирует или находит нужный ресурс и отправляет назад.

Популярные веб-серверы:

- **Nginx** — высокопроизводительный, асинхронный, доминирует на современном вебе. Часто используется как обратный прокси и балансировщик.
- **Apache HTTP Server** — классика, очень гибкий за счёт большого числа модулей.
- **IIS (Internet Information Services)** — Microsoft, под Windows-стек.
- **Caddy** — современный сервер с автоматическим TLS через Let’s Encrypt.
- **LiteSpeed**, **OpenLiteSpeed** — высокопроизводительные альтернативы.

Дополнительно веб-сервер обычно решает задачи **TLS-терминации** (расшифровка HTTPS), установки заголовков безопасности, **ограничения скорости (rate limiting)**, журналирования запросов и проксирования к бэкенду на другом порту.

Минимальный конфиг Nginx с обратным проксированием:

```nginx
server {
  listen 80;
  server_name example.com;

  root /var/www/html;
  index index.html;

  location /api/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}
```

### Прокси-сервер

**Прокси-сервер** — это промежуточный сервер между клиентом и целевым ресурсом, через который проходит трафик. Прокси может фильтровать, кэшировать, логировать, изменять запросы и ответы. Различают два основных типа.

**Прямой прокси (forward proxy)** работает на стороне клиента. Клиент сам настроен ходить через прокси. Применяется в корпоративных сетях для контроля сотрудников, фильтрации запрещённого контента, кэширования трафика, обхода ограничений или анонимизации. Примеры: Squid, Privoxy.

**Обратный прокси (reverse proxy)** стоит на стороне сервера. Клиент думает, что обращается к одному серверу, а на самом деле его запрос принимает обратный прокси и перенаправляет на нужный бэкенд. Это даёт сразу несколько преимуществ:

- **балансировка нагрузки** между несколькими экземплярами приложения;
- **кэширование** статических ответов;
- **TLS-терминация** в одной точке;
- защита бэкенда (он не виден напрямую из интернета);
- **WAF (Web Application Firewall)** — фильтрация атак;
- единая точка входа для микросервисов.

Популярные обратные прокси: Nginx, HAProxy, Traefik, Envoy. В облаках Amazon ALB/NLB, Google Cloud Load Balancer, Yandex Application Load Balancer выполняют ту же роль.

**Прозрачный прокси** — это такой прокси, о котором клиент не знает: его трафик перенаправляется на уровне сети (NAT). Часто используется ISP-провайдерами или в корпоративных шлюзах.

**Анонимизирующий прокси / VPN** скрывает реальный IP клиента, заменяя его адресом прокси-сервера.

### Что важно сказать в итоге

DNS превращает доменные имена в IP-адреса через распределённую иерархическую систему серверов с активным кэшированием. Клиент-серверная архитектура — основа вебa: клиент шлёт запросы, сервер обрабатывает и отвечает. Веб-сервер (Nginx, Apache) принимает HTTP-запросы и отдаёт контент, часто работая в паре с обратным прокси для балансировки, кэша и безопасности. Прокси-серверы добавляют гибкость на обоих концах: forward — на стороне клиента, reverse — на стороне сервера.
